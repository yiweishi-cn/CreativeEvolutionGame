Abstract

From one-shot generation to iterative, mechanic-aware creative improvement.

The original technical report presents CreativeGame as a multi-agent system that decomposes creative game generation into planning, code generation, testing, evaluation, reflection, and memory writing. The system uses a programmatic creativity proxy reward instead of relying primarily on subjective LLM judgment, and stores experience in a lineage-aware memory structure.

Based on subsequent project documents and code evolution, the current system is being reframed around a stronger principle: mechanics should not only explain a generated game after the fact, but should act as explicit design objects for the current version before code generation begins.

Latest interpretation of the project

  • The pipeline now aims to connect IntentSpec -> mechanic retrieval -> CurrentMechanicSet -> GameSpec -> code -> mechanic-grounded evaluation.
  • The key research question is no longer only "can AI generate games?" but "can AI learn to generate more creative games through structured iteration?"
  • The strongest evidence should come from visible lineage evolution, not isolated single outputs.

Implementation status

The planner now reads mechanic-library context, emits CurrentMechanicSet, and the frontend can visualize a mechanic trace for generated games.

Why the new layer matters

  • It distinguishes preserved mechanics from newly added mechanics.
  • It turns reward into planned structural realization rather than only post-hoc comparison.
  • It lets memory store mechanic deltas rather than only version summaries.

What was missing before

  • Mechanic library was mostly retrieval support and novelty baseline.
  • Game versions were still generated primarily from prompt interpretation.
  • Evaluator could compare versions, but not strongly compare plan vs implementation.

System Modules

Current technical structure in the codebase.

Planner

Retrieves mechanic context, proposes an intent-aligned design, and outputs CurrentMechanicSet.

Code Generator

Implements the planned version through staged generation: skeleton, feature, visual, and later refinement.

Tester + Evaluator

Checks runtime plausibility, extracts realized mechanics, and scores the generated artifact.

Reflector + Memory

Decides whether to continue, stores experiences in lineage memory, and can write back to the global archive.

Reward and Memory

What remains from the original report, and what is being strengthened.

Reward

The original report defines a weighted reward centered on structural mechanic change, novelty, LLM creativity, runtime playability, cosmetic penalties, and regression penalties.

The updated interpretation reframes reward toward a new question: did this version realize the intended mechanic change?

Lineage Memory

Memory remains lineage-scoped, which is still the right design for preserving game identity across versions and forks.

The newer design adds a stronger requirement that memory should store mechanic-level deltas and preservation decisions explicitly.

Mechanic Archive

The archive is no longer only a background novelty baseline. It is becoming an active design source for planner retrieval and generation guidance.

Implementation Status

What is grounded in code, what is in-progress, and what is still research debt.

Implemented

Already in the current code

  • Planner reads mechanic-library context (774 LLM-extracted entries).
  • CurrentMechanicSet is parsed from Planner output and stored per iteration.
  • CurrentMechanicSet forwarded to Evaluator and Reflector for planned-vs-realized comparison.
  • API returns mechanic trace and reward breakdown.
  • Gallery trace view visualizes mechanic-related planning and evaluation.
  • LLM extraction pipeline built: 738 structured archive entries with delta_A/T/O/F/K/W.
In Progress

Actively being strengthened

  • Wiring LLM extraction into live generation pipeline (replacing regex for mechanic_delta).
  • Planned-vs-realized mechanic comparison via realization_score in Evaluator.
  • Two-path memory write-back: lineage-local MemRecord + cross-lineage archive update.
  • Stronger demo framing around base game plus iterative checkpoints.
Open

Still missing for research-grade claims

  • Human pairwise evaluation of creativity improvement.
  • MemRL ablation and reward ablation studies.
  • Cleaner mechanic extraction beyond bounded regex coverage.
  • Proof that mechanic planning consistently drives better generation outcomes.

Limitations

Important caveats from the report and later discussion.

No human pairwise study yet

The report is explicit that there is still no human validation showing that later versions are systematically judged as more creative.

No MemRL ablation yet

The memory design is implemented, but its actual contribution has not yet been isolated experimentally.

Mechanic extraction: partially resolved

Archive building now uses LLM-based extraction (738 structured entries). The live generation pipeline still uses regex. Integration of LLM extraction into the generation loop is the next concrete task.

The key gap has shifted

The most important current challenge is no longer "can the system generate games?" but "can mechanics truly drive current-version generation?"

Future Work

Planned directions for strengthening the system.

The clearest next milestone is a stable mechanic-guided MVP: consolidate the schema around CurrentMechanicSet, ensure the planner emits it consistently across all generation paths, connect planned mechanics to evaluation via realization_score, and conduct human pairwise evaluation to validate that mechanic-aware planning produces measurably more creative outcomes.
