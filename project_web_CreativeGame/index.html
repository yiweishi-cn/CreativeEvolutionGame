CreativeEvoGame: Toward Self-Evolving Creative Game Generation.

CreativeEvoGame is a multi-agent pipeline where games self-evolve across iterations. A Planner retrieves mechanic context, emits CurrentMechanicSet, and drives Skeleton → Feature → Visual → Refinement generation. Each version is evaluated against its plan, reflected upon, and written into lineage memory — feeding the next iteration.

Abstract

From one-shot generation to mechanic-aware iterative improvement.

What the system does

CreativeGame decomposes game generation into planning, code generation, testing, evaluation, reflection, and memory writing. Each iteration produces a versioned game with a mechanic trace, reward breakdown, and lineage record.

Research question

Can AI generate more creative games through structured iteration and mechanic-aware planning? The pipeline connects IntentSpec → mechanic retrieval → CurrentMechanicSet → GameSpec → code → mechanic-grounded evaluation.

Pipeline

Five stages from intent to evaluated game.

01

Planner

Retrieves mechanic-library context and emits CurrentMechanicSet declaring what to preserve, add, remove, or recombine.

02

Code Generation

Skeleton → Feature → Visual → Refinement. Each sub-stage receives the planned mechanic set.

03

Tester

Static and optional browser validation. Failed runtime checks block reward.

04

Evaluator

Scores creativity and functionality (1–10), extracts realized mechanics, computes reward against CurrentMechanicSet.

05

Reflector + Memory

Writes mechanic-level deltas to lineage memory and optionally updates the global archive.

Before this version

  • Mechanic library was retrieval support and novelty baseline only.
  • Versions generated primarily from prompt interpretation.
  • Evaluator compared versions but not plan vs. implementation.

What CurrentMechanicSet adds

  • Explicitly distinguishes preserved from newly added mechanics.
  • Turns reward into planned structural realization, not post-hoc comparison.
  • Memory stores mechanic-level deltas instead of only version summaries.

Limitations

Current gaps and caveats.

No human evaluation yet

LLM self-evaluation inflates scores — the demo lineages assign C=8/F=8 to versions that are mechanically overcomplicated for human play. No controlled pairwise study exists yet.

No memory ablation yet

Memory is implemented but its contribution has not been isolated. It is unclear whether improvements come from memory or model capability alone.

Mechanic extraction partially resolved

Archive uses LLM-based extraction (738 entries). Live generation still uses regex for mechanic_delta. Full integration is the next concrete engineering task.

The key gap has shifted

No longer "can the system generate games?" but "can mechanic-aware planning consistently drive more creative outcomes than prompt-only generation?"

Discussion

Why reward decreases in later iterations.

Round 1 rewards are consistently 1.00. By round 3–4, reward drops to 0.55–0.80 in three of four lineages — despite stable LLM scores of C=8/F=8. This reflects a structural tension in the reward formula, not quality degradation.

structural_change decay

structural_change (weight 0.25) rewards added_mechanics and structural_change_score. In a "preserve + deepen" round, both approach zero — the formula cannot distinguish "no change" from "deliberate preservation."

novelty decay

relative_mechanic_novelty (weight 0.20) is Jaccard distance against the archive. Each version writes its mechanics back in — by round 3–4 the lineage's own prior versions fill the archive, driving novelty toward zero through self-competition.

cosmetic penalty misfire

When structural_change_score < 0.3, cosmetic_penalty = 0.4 is applied (weight 0.15). Planned visual-enhancement rounds like Flappy v3 trigger this, treating an intentional aesthetic pass as a cosmetic-only failure.

-0.3 baseline offset

The formula applies composite = raw × 2 − 0.3. A neutral iteration with no penalties still scores negatively. Late refinement rounds are structurally harder to score positively regardless of actual quality.

The reward signal and the intended creative strategy are misaligned: the pipeline aims at deepening lineages, but reward is calibrated for per-round novelty. A round-4 game that tightly couples three prior mechanics scores lower than a round-1 game that introduces one shallow mechanic. Two fixes: a cumulative mechanic coupling score that rewards integration of prior mechanics, and a lineage-relative novelty baseline that compares against the lineage's own history rather than the full archive.

Future Work

Next priorities.

Consolidate CurrentMechanicSet schema across all generation paths, connect planned mechanics to evaluation via realization_score, integrate LLM-based mechanic extraction into the live pipeline, and run human pairwise evaluation to validate that mechanic-aware planning produces measurably more creative outcomes than prompt-only baselines.
